\section{Overview of Contributions and Artifacts}

\subsection{Paper's Main Contributions}

\begin{description}
\item[$C_1$] Cross-vendor root-cause analysis: a unified methodology using backward dependence tracing to identify root-cause instructions from stalled instructions across NVIDIA, AMD, and Intel GPUs.
\item[$C_2$] Explicit memory dependency tracing: vendor-specific synchronization edges through AMD \texttt{s\_waitcnt} counters, NVIDIA hardware barrier bits, and Intel SWSB tokens.
\item[$C_3$] Cross-vendor evaluation: geometric-mean speedups of 1.73$\times$--1.82$\times$ across 21 workloads on three GPU platforms, with case studies demonstrating cross-vendor root-cause divergence.
\end{description}

\section{Artifact Identification}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\newartifact

\artrel

Artifact $A_1$ contains the LEO analysis tool, Docker-based evaluation infrastructure, benchmark workloads, profiling data, and optimization patches used to produce all experimental results in the paper (Tables~IV--V, Figure~5, and the five case studies in Section~VI: MASS3DEA, miniBUDE, QuickSilver, ML kernels (llama.cpp + HipKittens), and Kripke).

\artexp

For each of the 21 workloads on three GPU platforms, LEO should identify the dominant stall chain through backward dependence tracing. Applying the LEO-guided optimization (confined to the code region implicated by the top-ranked chain) should reproduce the per-kernel speedups reported in Table~IV within measurement noise ($\pm$5\% on NVIDIA/AMD, up to $\pm$9\% on Intel PVC due to higher platform variability).
The single-dependency coverage (Figure~5) should show improvement from LEO's analysis pipeline on NVIDIA and AMD, with Intel coverage remaining stable due to compiler-verified SWSB edges.
The LLM diagnostic study (Table~V) evaluates Gemini 3.1 Pro on 15 RAJAPerf CUDA kernels ($k{=}5$ trials, temperature 0.7), comparing three diagnostic contexts: code only (C), code plus raw stall counts (C+S), and code plus LEO's structured root-cause analysis (C+L(S)).

\arttime

\begin{itemize}
\item \textbf{Artifact Setup:} 30--60 min (Docker image build per vendor).
\item \textbf{Artifact Execution:} 2--4 hours total (profiling + LEO analysis across 21 workloads $\times$ 3 vendors; benchmarking original vs.\ optimized kernels).
\item \textbf{Artifact Analysis:} 15 min (SDC collection script, speedup comparison scripts).
\end{itemize}

\artin

\artinpart{Hardware}

Three GPU platforms are required to reproduce all results:
\begin{itemize}
\item \textbf{NVIDIA:} GH200 480GB (sm\_90, Hopper), CUDA 12.8, driver 570+
\item \textbf{AMD:} MI300A (gfx942, CDNA3 APU), ROCm 6.3
\item \textbf{Intel:} Data Center GPU Max 1100 (Ponte Vecchio, Xe-HPC), Level Zero 1.x, oneAPI 2025.1+
\end{itemize}
Partial reproduction on a single vendor is possible; the evaluation scripts support per-vendor execution.

\artinpart{Software}

\begin{itemize}
\item \textbf{LEO} (Python 3.11+, uses \texttt{uv} package manager): \url{https://github.com/yuninxia/leo} (private; public upon acceptance)
\item \textbf{HPCToolkit} (branch \texttt{feature/hidden-stall-metrics}): \url{https://gitlab.com/yuningxia/hpctoolkit.git}
\item \textbf{hpcanalysis} (C++ database reader with pybind11): included as git submodule
\item \textbf{RAJAPerf} v2024.07.0: \url{https://github.com/LLNL/RAJAPerf}
\item \textbf{Docker} 24.0+: all evaluation runs inside vendor-specific containers
\item \textbf{nvdisasm} (CUDA Toolkit), \textbf{llvm-objdump} (ROCm), \textbf{GED} (Intel GTPin 4.5.0): GPU disassemblers, included in Docker images
\item \textbf{Gemini 3.1 Pro API} (Google Cloud): required only for Table~V reproduction. Needs internet connectivity and valid API credentials. Table~IV and Figure~5 do not require LLM access.
\end{itemize}

\artinpart{Datasets / Inputs}

The 21 workloads evaluated in Table~IV consist of 15 RAJAPerf kernels plus 6 HPC/ML applications. HipKittens RMSNorm is additionally studied in Section~VI (not part of Table~IV).

\begin{itemize}
\item \textbf{RAJAPerf kernels (15):} \texttt{Base\_CUDA}, \texttt{Base\_HIP}, and \texttt{Base\_SYCL} variants with default problem sizes, included in RAJAPerf repository. Kernels: MASS3DEA, LTIMES\_NOVIEW, LTIMES, 3MM, 2MM, GEMM, PRESSURE, ENERGY, FIR, ZONAL\_ACCUM\_3D, VOL3D, DEL\_DOT\_VEC\_2D, DIFFUSION3DPA, CONVECTION3DPA, MASS3DPA.
\item \textbf{HPC applications (4):} LULESH, miniBUDE, XSBench, Kripke --- all open-source with default inputs.
\item \textbf{ML applications (2):} llama.cpp (Qwen2.5-1.5B, Q4\_K\_M quantization), QuickSilver.
\item \textbf{Case study (not in Table~IV):} HipKittens RMSNorm (AMD-only, hand-tuned ML kernel).
\item \textbf{Pre-collected profiling data:} HPCToolkit measurement databases for all 21 workloads $\times$ 3 vendors, archived in the artifact repository. These enable LEO analysis without access to GPU hardware.
\end{itemize}

\artinpart{Installation and Deployment}

\begin{enumerate}
\item Clone LEO repository with submodules: \texttt{git clone --recursive}
\item Install Python dependencies: \texttt{uv sync}
\item Build Docker images (one per vendor): \texttt{bash scripts/evaluation/build\_containers.sh}
\item Alternatively, use the universal analysis container (\texttt{leo-base-universal}) for LEO-only analysis on pre-collected data (no GPU required).
\end{enumerate}

\artcomp

The evaluation workflow consists of four tasks:

\begin{description}
\item[$T_1$: Profile] Run \texttt{hpcrun} with PC sampling on each workload inside vendor-specific Docker containers. Output: HPCToolkit measurement databases.
\item[$T_2$: Analyze] Run LEO (\texttt{scripts/analyze\_benchmark.py}) on each measurement database. Output: ranked stall chains, dependency graphs, SDC metrics.
\item[$T_3$: Benchmark] Run original vs.\ optimized kernels (\texttt{benchmarks/rajaperf-h100/run\_compare.sh}). Output: per-kernel speedup measurements.
\item[$T_4$: Collect] Run collection scripts (\texttt{scripts/collect\_sdc.sh}, \texttt{scripts/time\_analysis.sh}). Output: SDC coverage data (Figure~5), analysis timing data.
\end{description}

Dependencies: $T_1 \rightarrow T_2 \rightarrow T_4$; $T_1 \rightarrow T_3$. Tasks $T_2$ and $T_3$ can run in parallel after $T_1$.

For pre-collected data (included in artifact), $T_1$ can be skipped entirely, and $T_2$--$T_4$ can be executed on any x86-64 machine using the universal Docker container without GPU hardware.

\textbf{Parameters:}
\begin{itemize}
\item RAJAPerf: default problem sizes, kernel-specific repetition counts with internal warmup.
\item Case study applications: 10 independent runs per configuration, mean speedup reported.
\item LEO analysis: \texttt{--top-n 1} for per-kernel analysis, \texttt{--top-n 5} for whole-program analysis.
\end{itemize}

\artout

Raw experimental results are processed by:
\begin{enumerate}
\item \texttt{scripts/collect\_sdc.py}: Extracts per-kernel SDC values from LEO output, generates the data for Figure~5.
\item \texttt{scripts/plot\_sdc\_paper.py}: Produces Figure~5 (SDC coverage chart) from collected data.
\item \texttt{benchmarks/rajaperf-h100/postprocess.py}: Computes speedup ratios from raw timing CSV files for Table~IV.
\item \texttt{scripts/dump\_paper.py}: Generates a full-text markdown dump for review.
\end{enumerate}
